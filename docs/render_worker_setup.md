# Render Worker Setup (Background Worker)

## Purpose
`apps/worker` is the async runtime for AI Tutor. It handles queue and scheduler work that should not block web requests.

Current responsibilities:
1. Claim queued jobs from Supabase using `claim_agent_jobs`.
2. Process job lifecycle (`claimed -> running -> completed|queued(retry)|failed`).
3. Write `agent_job_events` for UI/SSE visibility.
4. Process artifact/chat side effects (build logs, project state, skill evidence, token usage).
5. Run recurring schedulers for news refresh + daily update generation.

## Why this should run on Render
1. It is a long-running process (interval loop), not an HTTP API.
2. Vercel serverless functions are request-driven; they are not ideal for persistent polling loops.
3. Render Background Worker keeps one durable process alive continuously.

## Render Service Type
Use **Background Worker**.

## Build / Start Commands
From repo root:

- Build command:
`pnpm install --frozen-lockfile && pnpm --filter @aitutor/worker build`

- Start command:
`node apps/worker/dist/index.js`

Alternative (less strict) start command:
`pnpm --filter @aitutor/worker dev`

## Required Worker Environment Variables (Render)
Required:
1. `SUPABASE_URL`
2. `SUPABASE_SERVICE_ROLE_KEY`

Optional but recommended:
1. `WORKER_ID` (default autogenerated)
2. `CLAIM_LIMIT` (default `5`)
3. `WORKER_POLL_MS` (default `2500`)
4. `SCHEDULER_POLL_MS` (default `60000`)
5. `DEFAULT_USER_ID` (default `user_test_0001`)

Notes:
1. Worker reads `SUPABASE_SERVICE_KEY` as a fallback alias for `SUPABASE_SERVICE_ROLE_KEY`.
2. Current worker implementation does **not** directly call OpenAI/Resend; those keys are not required for the worker process today.

## Vercel `RENDER_API_KEY` vs Worker Runtime
`RENDER_API_KEY` in Vercel is only for deployment automation/management calls to Render API.
It is **not** required by worker runtime itself.

If Render API calls return `401 Unauthorized`, the key is invalid or revoked. Replace it in Vercel.

## Production Operating Guidance
1. Run one worker instance for now (avoids duplicate scheduler writes).
2. Confirm worker logs show periodic lines like:
   - `[worker] job completed: ...`
   - `[worker] news refreshed: ...`
   - `[worker] daily update sent`
3. If provider outage occurs, retries are scheduled with backoff and fail-state events.

## Quick Validation After Deploy
1. Trigger a chat/artifact action from app.
2. Confirm rows appear in `agent_jobs` and `agent_job_events`.
3. Confirm job transitions to `completed` (or `failed` with `last_error_code`).
4. Confirm new `build_log_entries` and `project_artifacts` rows for artifact jobs.
